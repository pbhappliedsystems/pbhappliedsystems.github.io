Which Model, Which Hardware,
Which Behavior

Perplexity and leaderboards answer the wrong question for anyone deploying a language model as an agent. quant_eval measures agent-relevant behavior directly and resolves it per capability across full-weight and quantized checkpoints — and its load-bearing discipline is telling a genuine capability loss apart from a mere scoring artifact. This is the engineering blog; the complete technical whitepaper is available as a downloadable PDF.

Patrick Hill, M.S. — Founder & Principal AI/ML Systems Architect PBH Applied Systems, LLC Oklahoma City  ·  July 2026
Download the quant_eval Whitepaper (PDF) → Try the Evaluated Models Live

Open-weight language models are usually selected on two kinds of evidence: an aggregate benchmark leaderboard, and — for local deployment — whether a small quantized checkpoint fits on the hardware at hand. Neither answers the question a practitioner building an agent actually asks. A model can rank well on both and still fail in production because it emits JSON that does not validate against a schema, names a tool under the wrong key, or loses track of state between turns. And quantization compounds the problem: a 4-bit checkpoint may preserve some agent-relevant behaviors while silently losing others, so a single "how much did it degrade" number is not actionable — the honest answer is "it depends which behavior you need."

quant_eval is a behavioral evaluation harness built to close that gap. It is behavioral in that every score derives from whether the model did the task correctly — valid JSON against a declared schema, the right tool with the right arguments, a correct multi-step plan with correct stop semantics — rather than from token-level likelihood. It is capability-resolved in that it reports a profile across eight distinct test families rather than a scalar, so a model that keeps tool calling but loses multi-step planning is described as exactly that. And it is quantization-aware in that its native mode runs the full-weight (16-bit) model and the quantized model side by side on identical seeded fixtures, isolating what quantization costs per capability. This post documents version 7.21 and grounds it in a worked example: one 12-billion-parameter model evaluated at three precisions through the same harness.

01 — The Test Families

Eight behaviors an agent runtime depends on

Each model is evaluated against up to eight families of cases, each scored by a dedicated evaluator against golden-oracle expectations. json is single-step structured output under explicit constraint rules. json_multistep is multi-step planning with a self-check trace and oracle verification of the final state — the hardest family and the most sensitive discriminator of quantization degradation. stateful_followup is two-turn state tracking. mixed_brief_json is a hybrid natural-language-plus-JSON response. mcq is multiple-choice extraction under a one-token budget. toolcall and toolcall_only cover tool dispatch — the first scored in two stages (parse-and-validate, then final computed value), the second a bare schema-only call checking tool name and argument correctness. fuzz is property-based regression over generated cases for stability. Each is scored as its own family because each can degrade independently under quantization.

02 — Architecture

One interface, four backends

The framework spans four inference backends behind a single interface: 16-bit inference via Hugging Face Transformers (the full-weight baseline), GGUF served through llama.cpp, NormalFloat-4 via bitsandbytes, and W4A16 compressed-tensors served through vLLM. All four expose the same chat_generate method, so the batteries that drive them are backend-agnostic — the runner's identity is recorded in the manifest and every output row, but the execution and scoring code never branches on it.

The central design decision is one adapter per model family. Each adapter answers whether it can handle a given model, returns a routing decision with a human-readable reason, reports capability flags, and constructs the full-weight and quantized runners. Adapters live in an explicit registry whose insertion order is the resolution policy, which makes routing deterministic and auditable rather than implicit — and it resolves genuine collisions, such as a directory named Qwen2.5-32B-Instruct_4bit_... that a generic 32B adapter would otherwise claim before the 4-bit adapter could. Routing then splits on a single flag, requires_gguf(): the GGUF lane converts a 16-bit checkpoint to F16 GGUF and quantizes it, while the GGUF-less lane takes a pre-quantized on-disk directory (NF4 or compressed-tensors) as the artifact under test. Everything downstream of runner construction — battery execution, scoring, manifest, reports — is shared between the two lanes without modification.

A deliberate separation runs through the design: batteries execute, evaluators score. A battery drives a runner over a family's cases and returns raw rows with no scoring logic; an evaluator then consumes each row and produces the correctness signals. The same execution can be re-scored, and the scoring logic is auditable in one place per family, independent of how the output was produced.

03 — Determinism

The golden oracle, hash-pinned

Comparability across models and precisions requires that every model see the same inputs under the same decoding conditions. The framework fixes the seed at 42, uses greedy decoding for scoring, and applies an identical system message and identical prompts to the full-weight and quantized runs. Correctness is defined by a deterministic oracle rather than a reference model: for the planning families, a first-fit oracle computes the correct plan, the correct per-step feasibility checks, and the correct final state for every case. Because the oracle is deterministic and the fixtures are fixed, the same case has the same correct answer for every model, and differences in score are attributable to the model, not the harness.

The fixture set is versioned and hash-pinned. Each run writes a content-addressed copy of the fixtures into the run directory and records the SHA-256 in the manifest — the runs discussed here used golden_oracle_fixtures_v7_21 with SHA-256 prefix 6d71a0b9 — and a drift check compares the oracle-derived expectations against the committed fixtures, so a silent change to the ground truth cannot pass unnoticed. The published hash serves integrity verification without exposing the fixture contents themselves.

04 — The Load-Bearing Discipline

Artifact versus genuine failure

The most consequential methodological commitment in quant_eval is the separation of harness artifacts from genuine capability failures. A harness artifact is a scored failure caused by output-format drift or decoding behavior rather than by the model getting the task wrong — and treating artifacts as genuine failures would systematically understate model capability, worse still unevenly across backends whose decoding conventions differ. The framework maintains a named taxonomy of recurring artifacts and neutralizes each centrally: role-label leakage from a chat template, end-of-sequence contamination, tool-argument key drift (args vs. arguments vs. params), tool-name key drift (tool_name, action, function), and the one-token budget tripping a reasoning model's thinking preamble.

A failure that persists across re-runs and across backends is a genuine capability result. A failure that a correct normalization removes — and that a second backend does not reproduce — is an artifact.

Normalization is centralized in one shared evaluator module, so a single fix applies to every family and every backend rather than being reimplemented per adapter. Genuine findings are held to a re-run-stability standard before they are reported. This is the framework's most load-bearing move, because the credibility of the entire catalog rests on classifying a scoring failure correctly — as the worked example shows.

05 — Worked Example

One model, three precisions

To make the methodology concrete, a single base model — Mistral-Nemo-Instruct-2407, a 12-billion-parameter, function-calling-trained model under Apache 2.0 — was evaluated at three precisions: FP16 via Transformers (the baseline), Q4_K_M GGUF via llama.cpp, and W4A16 AWQ via vLLM. The first two ran as a single side-by-side run producing a per-capability degradation delta; the third was a later quant-only run on a different backend, used to corroborate the artifact analysis. All used the same fixtures, seed 42, and the fast_gate profile on an RTX 4090.

The shape of the result is the point. Quantization preserved single-step JSON, property-based regression, two-turn state tracking, the hybrid format, and stage-one tool-call formatting. It genuinely degraded multi-step planning (one hard case collapsed to no parseable plan, and the self-check reasoning became less consistent), multiple-choice accuracy (one item flipped from a correct B to an incorrect A), and tool-call final-value computation (the call still parsed and validated, but the model emitted 123 where the correct sum of 10 and −4 is 6). Reporting toolcall in two stages is what makes that legible: tool calling did not simply "break" — it retained correct call structure while losing final-value reliability.

The most instructive line is toolcall_only, which fell from a perfect 1.000 to 0.000. Taken at face value, that says the quantized model lost the ability to call a tool. The raw outputs say otherwise: the tool name was correct in every case, and the failure was entirely in the arguments — the quantized model emitted them under a recognized params alias but with a trailing extra brace. Two independent lines of evidence establish this as an artifact. First, centralized normalization recovers it: re-scoring the exact quantized outputs extracts the first balanced JSON object, discards the stray brace, resolves the alias, and returns the correct arguments. Second, a second backend does not reproduce it: the later AWQ run on vLLM scored toolcall_only at 1.000. Had the framework scored the artifact as genuine, it would have published that a function-calling model cannot call functions after quantization — a conclusion contradicted by the model's own correct tool name and operands.

The AWQ profile agrees with the Q4_K_M profile on every preserved family and on the fragile one (json_multistep at 0.400, mcq at 3/5), and differs on exactly the two lines the artifact analysis predicted: toolcall_only recovers to 1.000 and toolcall to a clean bucket of 11.000, because the vLLM path decoded the final value without contamination. One base model, two quantization algorithms, two serving stacks, one harness — and a consistent capability story.

06 — Cost & Footprint

What the trade is set against

The behavioral profile is only half of a deployment decision; the other half is cost, and the run manifest records it directly. The F16 GGUF was about 24.5 GB and the Q4_K_M GGUF about 7.48 GB — a roughly 3.3× reduction that moves the model from the ceiling of a 24 GB GPU to an 8 GB one. Mean per-case latency moved commensurately: about 30.24 s under FP16 Transformers, 1.42 s under Q4_K_M llama.cpp (a roughly 21× speedup), and 0.936 s under AWQ on vLLM. These are eval-harness measurements under a fixed configuration, not production throughput benchmarks, but they quantify the trade the profile is set against: for this model, the preserved families are available at a fraction of the footprint and latency, and the price is paid specifically in multi-step planning and multiple-choice reliability.

07 — Selection Layer

A decision instrument, not a rank

The output of a quant_eval run is not a rank; it is a per-model, per-backend capability profile with honest cautions. For the worked example, that profile reads: deploy with confidence for tool dispatch, two-turn stateful interaction, hybrid output, and single-step JSON; deploy with an external validation loop for multi-step planning; verify on your own option distributions before relying on multiple-choice; and, in a serving stack that does not normalize tool wrappers, enforce the canonical keys through the system prompt. That is a set of engineering instructions, not a leaderboard position. When such profiles are produced for many models across the backends a team can actually run, they compose into a selection layer: given a required behavior and a hardware budget, the profiles narrow the field to the checkpoints that measurably support that behavior at that budget — which is exactly the layer behind the evaluated models in the live demo.

08 — Limitations & Reproducibility

What the numbers do and do not support

The worked example supports strong claims about the framework — that it measures per-capability degradation, separates artifacts from genuine failures, and does so identically across three backends — because those claims are about the instrument and are demonstrated by its behavior on real runs. It supports only provisional claims about the model, because the runs used the fast_gate screening profile with small per-family case counts. A pass rate of 0.400 on five json_multistep cases means two of five passed; a single additional passing case moves it to 0.600. The framework marks every such verdict provisional with an explicit "n < 10" annotation, precisely so a screening result is not mistaken for a definitive measurement. The reference node is a single 24 GB GPU, so models whose 16-bit weights exceed that budget are evaluated quant-only, yielding pass rates but no degradation delta; several families are grounded in one deterministic planning domain, which buys an unambiguous oracle at the cost of breadth; and scoring is greedy at a fixed seed, characterizing most-likely behavior rather than sampling variability.

Every run is designed to be independently verifiable from its recorded artifacts. The versioned, hash-pinned fixtures; a run_manifest.json recording the adapter and its reason, the runner and role, the quantization type, the seed, and the byte sizes and SHA-256 hashes of the model artifacts; a per-case comparison CSV containing the raw output, timing, and every scored signal; and regenerable derived reports — together these let a third party recompute the reported pass rates from the CSV, verify the artifacts against the recorded hashes, and re-derive the reports. That is the standard of reproducibility the framework's credibility depends on. The contribution is not a benchmark number but a disciplined, reproducible, backend-agnostic method for producing the evidence a deployment decision actually requires.
